What Is AutoResearch?

The Problem With How We Optimize Today

Here is how most optimization works. A human forms a hypothesis. They design an experiment. They run it. They wait hours, days, sometimes weeks for results. They analyze the data. They form a new hypothesis. Repeat.

One person. One experiment at a time. Serial processing in a parallel world.

The bottleneck is not compute. It is not data. It is the human in the loop, context-switching between tasks, sleeping, eating, attending meetings. The actual optimization work happens in short bursts between everything else.

Karpathy's Insight

Andrej Karpathy, former head of AI at Tesla and co-founder of OpenAI, asked a simple question: what if an AI agent ran the entire optimization loop autonomously?

Not "assist with" the loop. Not "suggest hypotheses." Run the whole thing. Form hypotheses, modify code, execute experiments, evaluate results, keep what works, revert what doesn't. Over and over. While you sleep.

He built it in 630 lines of Python. MIT licensed. Published it on GitHub. Within days: 41,700 stars.

The project is called AutoResearch, and the core idea is deceptively simple: give an AI agent a metric to optimize, a file it can modify, and a time budget. Then walk away.

How It Works: The Framework

AutoResearch has three components. That's it. Three files that create an autonomous optimization engine.

The Three Components

Component File Who Controls It Purpose
Infrastructure prepare.py Human (fixed) Data prep, tokenizer training, evaluation utilities
Experiment Surface train.py AI Agent (modifiable) Complete model, optimizer, training loop
Instructions program.md Human (maintained) Guides agent behavior, defines constraints

prepare.py handles everything the agent should never touch: downloading datasets, training tokenizers, building evaluation pipelines. This is your foundation. Locked down. Deterministic.

train.py is the playground. This single file contains the complete training pipeline: model architecture, optimizer configuration, learning rate schedules, batch sizes, everything. The agent can modify any part of it.

program.md is the skill definition for the AI. It tells the agent what metric to optimize, what constraints to respect, and how to approach experimentation. Think of it as a detailed brief you would give a new hire.

The Loop

Here is what happens when you start a run:

  1. The agent reads program.md for instructions and context.
  2. The agent forms a hypothesis. Example: "Increasing the learning rate by 10% during the warmup phase may reduce validation loss."
  3. The agent modifies train.py to implement the change.
  4. The experiment runs for exactly 5 minutes. Fixed budget. No exceptions.
  5. The agent evaluates the result: validation bits per byte (val_bpb). Lower is better.
  6. If the metric improved: the change is kept. If not: the change is reverted.
  7. The agent forms a new hypothesis informed by what it just learned.
  8. Repeat.

At roughly 12 experiments per hour, that is 100 experiments in an overnight run. 100 hypotheses formed, tested, and evaluated without a single human interaction.

Beyond ML: The AutoResearch Pattern for ANY Metric

Here is where it gets interesting for people who don't train neural networks.

The AutoResearch pattern is not about machine learning. It is about autonomous, iterative optimization. The pattern applies anywhere you have three things:

  1. A measurable metric you want to improve
  2. Variables you can change that influence that metric
  3. Fast enough feedback loops to evaluate changes

That describes almost every business optimization problem.

Scenario 1: Cold Email Optimization

ElementDetails
MetricReply rate
VariablesSubject line, opening sentence, CTA, send time
Feedback speed24-48 hours per batch

The loop: Generate 10 email variants using an LLM. Send each to a test segment of 50-100 recipients. Wait 48 hours. Measure reply rates. Analyze what worked: was it the question-based subject line? The personalized opener? The soft CTA? Feed findings into the next generation of variants.

Expected trajectory: Most cold email campaigns start at 2-4% reply rates. After 4-6 weeks of autonomous iteration (5-8 cycles), teams consistently reach 8-12%.

Scenario 2: Landing Page Conversion

ElementDetails
MetricConversion rate (visitor to signup/purchase)
VariablesHeadlines, subtext, CTAs, social proof placement, layout
Feedback speed200-500 visitors per variant

The loop: Generate page variants with different headlines, value propositions, and CTA copy. Deploy via A/B testing infrastructure. Collect data until statistical significance (minimum 200 visitors per variant). Analyze winners. Generate the next round of variants informed by what worked.

Expected trajectory: 15-40% improvement in conversion rate over 8-12 weeks. The compounding effect is significant: a 3% baseline conversion moving to 4.2% is a 40% revenue increase from the same traffic.

Scenario 3: Paid Ad Copy

ElementDetails
MetricClick-through rate (CTR) and cost per acquisition (CPA)
VariablesHeadlines, descriptions, creative formats, audience targeting
Feedback speed24-72 hours with sufficient budget

The loop: Generate ad copy variants. Deploy via platform APIs (Google Ads, Meta Ads). Set budget caps and performance floors as guardrails (pause any variant spending more than $X with CTR below Y%). Measure after minimum impressions threshold. Iterate.

The guardrails are critical here. Unlike a training loop where a bad experiment costs 5 minutes of compute, a bad ad variant costs real money. Build in hard stops.

Scenario 4: Newsletter Engagement

ElementDetails
MetricOpen rate, click-through rate
VariablesSubject lines, preview text, send timing, content format
Feedback speed24-48 hours per send

The loop: Generate subject line and preview text variants. A/B test across subscriber segments. Measure opens and clicks at the 48-hour mark. Analyze patterns (do questions outperform statements? does personalization help?). Feed learnings into the next send.

With a 2-3x/week send cadence, you can run 8-12 experiments per month. Over a quarter, that is 24-36 tested hypotheses on a single channel.

Scenario 5: Pricing and Offer Framing

ElementDetails
MetricRevenue per visitor, average order value (AOV)
VariablesOffer structures, discount presentations, upsell sequences, pricing page layout
Feedback speedVaries by traffic volume

The loop: Test different framings of the same offer. "Save $50" vs. "Save 25%" vs. "Get 3 months free." Test upsell timing, bundle structures, and anchor pricing. Measure revenue per visitor, not just conversion rate (a lower conversion at a higher price point can win).

This is the highest-impact application of the pattern. Small pricing improvements compound directly into revenue.

Implementation: How to Set Up Your Own AutoResearch Loop

For ML/Technical Users

If you want to run Karpathy's original AutoResearch on a model training task:

Bash 
# Step 1: Clone the repository
git clone https://github.com/karpathy/autoresearch
cd autoresearch

# Step 2: Install dependencies
uv sync

# Step 3: Prepare your data
uv run prepare.py

# Step 4: Test a manual training run
uv run train.py

# Step 5: Point your AI agent at program.md
# Use Claude Code, OpenAI Codex, or any capable coding agent
# The agent reads program.md and begins the loop

# Step 6: Let it run overnight
# ~12 experiments/hour = ~100 experiments in 8 hours

Requirements: a GPU with sufficient VRAM for your model, Python 3.10+, and an AI coding agent with file system access.

For Business Users: The 4-Week Ramp

You don't need to run the original repo. You need to build the pattern for your domain. Here is a realistic timeline.

Week-by-Week Ramp

Week 1: Variant Generation
Pick one metric and one variable. Email subject lines are a good starting point. Set up an LLM prompt that generates 10 variants given your product, audience, and previous winners. Test the generation quality manually. Refine the prompt until variants are usable without human editing.

Week 2: Automated Performance Retrieval
Connect to your platform's API (Mailchimp, HubSpot, Google Analytics, whatever you use). Build a script that pulls performance data for each variant. Automate the "measure" step so no human needs to log in and check dashboards.

Week 3: Programmatic Deployment
This is the hard part. Set up API connections to deploy variants without manual intervention. For email, this means programmatically creating A/B test campaigns. For landing pages, this means deploying variants via your CMS or testing tool API. For ads, this means using the ad platform's API.

Week 4: The Full Loop
Connect generation, deployment, measurement, and analysis into one pipeline. Add autonomous scheduling so it runs on a cron job or event trigger. The loop runs itself: generate variants, deploy, wait for data, analyze, generate the next round.

Minimum Technical Components

Component Purpose Examples
Data source access Pull performance metrics Platform APIs, analytics APIs
Content generation Create test variants Claude API, GPT API, any LLM
Deployment automation Push variants live Platform APIs, CMS APIs, ad APIs
Autonomous scheduling Run without triggers Cron jobs, n8n workflows, GitHub Actions
Findings documentation Compound learnings Markdown logs, database entries

The findings documentation piece is often overlooked. Each cycle must inform the next. Without it, you are running random experiments, not compounding intelligence.

Critical Success Factors

1. One Metric, One Variable

Do not test everything at once. If you change the subject line AND the send time AND the opening sentence, you have no idea which change drove the result. Isolate variables. Test one thing per cycle.

This is the single most common mistake. Resist the temptation to "test more" per cycle. Speed comes from fast cycles, not from cramming more variables into each one.

2. Statistical Rigor

Minimum sample sizes before declaring a winner:

Channel Minimum Per Variant
Email (open rate)50 recipients
Email (reply rate)100 recipients
Landing page200 visitors
Paid ads1,000 impressions
Pricing tests500 visitors

Acting on small samples is worse than not testing at all. You will "learn" patterns that are just noise, then optimize toward randomness.

3. Feedback Loop Closure

Previous learnings MUST inform the next round of variants. This is what separates AutoResearch from random A/B testing.

Bad: Generate 10 random variants each cycle.
Good: Feed the top 3 winners and bottom 3 losers from last cycle into the generation prompt. Ask the LLM to analyze what worked, hypothesize why, and generate variants that push the winning patterns further.

4. Extended Runway

Meaningful gains emerge after 4-6 weeks. The first 1-2 weeks are calibration: the system is learning what moves the needle at all. Weeks 3-4 are where compounding kicks in as the agent starts combining multiple winning elements.

Do not judge the system by week 1 results.

5. Human Oversight

Check in on findings every 3-5 cycles. Look for:

  • Errors in data collection (are metrics being measured correctly?)
  • Overfitting to a narrow segment (does a "winning" variant only work on one audience?)
  • Ethical drift (is the AI generating manipulative copy to hit the metric?)
  • Metric gaming (is the system finding loopholes rather than genuine improvements?)

Autonomous does not mean unsupervised.

6. Cross-Channel Compounding

What works in email informs ad copy. What works in ads informs landing pages. What works on landing pages informs the sales pitch.

Once you have findings from one channel, feed them into variant generation for adjacent channels. The insight "question-based headlines outperform statement-based headlines by 23%" applies far beyond the channel where you discovered it.

Common Mistakes

1. Testing too many variables simultaneously.
You cannot learn what worked when everything changed. One variable per cycle. Always.

2. Acting on insufficient sample sizes.
50 data points is not a pattern. It is noise. Wait for statistical significance. The automation is patient even if you are not.

3. Optimizing the wrong metric.
Open rates are a vanity metric if they don't correlate with revenue. Make sure your target metric connects to a business outcome. Optimize for reply rate, not open rate. Optimize for revenue per visitor, not page views.

4. Not feeding learnings back into the loop.
Without compounding intelligence, you are running isolated experiments. Each cycle must build on the last. This requires explicit documentation of findings and structured prompts that incorporate previous results.

5. Stopping too early.
The biggest gains come from compounding small improvements over time. A 3% improvement per cycle across 12 cycles is a 42% total improvement. But you only see the 3% if you look at a single cycle. The magic is in the persistence.

The Big Picture

"All LLM frontier labs will do this. It's the final boss battle." -- Andrej Karpathy

His vision goes beyond a single agent running a single optimization loop. The endgame is multiple agents exploring different optimization paths in parallel. One agent testing learning rate schedules while another explores architecture changes while a third experiments with data augmentation.

"The goal is not to emulate a single PhD student. It is to emulate a research community of them." -- Andrej Karpathy

This reframes the role of the human. You are no longer the researcher. You are the research director. You set the objectives, define the constraints, allocate resources, and review findings. The agents do the grunt work: the hundreds of small experiments that map the optimization landscape.

And this is not limited to ML researchers in San Francisco. Anyone with a measurable metric and controllable variables can run this pattern. A solo founder optimizing cold outreach. A marketing team iterating on ad copy. A product team refining onboarding flows.

The pattern is universal. The tooling is open source. The only question is whether you set up the loop or keep running experiments one at a time.

Quick Reference: AutoResearch in 60 Seconds

The Complete Loop

  1. Pick ONE metric to optimize
  2. Identify the variables that influence it
  3. Set up automated measurement (API access to your data)
  4. Set up automated variant generation (LLM prompt)
  5. Set up automated deployment (platform APIs)
  6. Connect the loop: generate, deploy, measure, analyze, repeat
  7. Run for 4-6 weeks minimum
  8. Review findings every 3-5 cycles
  9. Feed winning patterns into adjacent channels

The loop does the work. You set the direction.
